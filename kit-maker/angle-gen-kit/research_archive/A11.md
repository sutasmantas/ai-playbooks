# A11 — github-broad-discovery

## Key findings

**Finding 1: Topic tags have critically low adoption — topic-only search misses the vast majority of relevant repositories**
- Claim: Only 0.001% of repositories and 0.04% of data assets on GitHub can be found through Collections, Topics, and standard code search. Topics are free-form, so the same concept is tagged under different labels (e.g., `llm-agent` vs `llm-agents` vs `agent-framework`), and most maintainers never add topics at all. GitHub had to build an ML-based topic-suggestion system to address adoption gaps, but opt-in rates remain low.
- Source: arxiv:2306.06191 (Open Data on GitHub, 2023); GitHub Blog "Topic Suggestions for Millions of Repositories" (github.blog/engineering/user-experience/topics/); Garousi et al. MLR guidelines
- Confidence: HIGH
- Verify note: The 0.001% figure comes from the Open Data on GitHub study (arxiv:2306.06191) specifically for data asset discoverability, not general code. The principle (low topic adoption) is confirmed by GitHub's own blog announcing a topic-suggestion ML system to address the gap. Scope: applies broadly to code repos; the exact percentage is domain-specific.

**Finding 2: Star counts are degraded quality signals — 16.66% of repos with 50+ stars had fake-star activity by mid-2024**
- Claim: A CMU/NCSU/Socket joint study (StarScout, ICSE 2026, arxiv:2412.13459) analyzed 20 TB of GitHub metadata and identified approximately 6 million suspected fake stars across 18,600 repositories. By mid-2024, 16.66% of all repositories with 50 or more stars were involved in fake-star campaigns. AI/LLM repositories were the largest non-malicious category affected (177,000 suspected fake stars). Detection signals: star-to-fork ratio below 50/1k on high-star repos; synchronized burst-starring patterns; stargazers with zero followers and no other repos.
- Source: arxiv:2412.13459 (Wermke et al., ICSE 2026, CMU/NCSU/Socket); https://arxiv.org/html/2412.13459v2
- Confidence: HIGH
- Verify note: Peer-reviewed at ICSE 2026, methodology publicly documented. The 16.66% figure applies to repos with 50+ stars as of mid-2024 — the threshold matters. AI/LLM repos are specifically named as the highest non-malicious fake-star category, making this directly relevant for research use of stars as quality signals in that domain.

**Finding 3: README content is not searched by default — `in:readme` must be explicit, and most researchers miss it**
- Claim: GitHub's repository search defaults cover name, description, and topics only. README content — where most projects describe their methodology, linked papers, and technical claims — requires the explicit qualifier `in:readme`. This is non-obvious: the GitHub community forums show recurring complaints from researchers who assumed README was included. For research topics where terminology appears in documentation but not in repo names or topic tags, omitting `in:readme` produces structurally incomplete results.
- Source: GitHub Docs "Searching for repositories" (docs.github.com/en/search-github/searching-on-github/searching-for-repositories); GitHub Community Discussion #59308; freecodecamp.org/news/github-search-tips/
- Confidence: CONSENSUS
- Verify note: GitHub's official documentation confirms this behavior. The community thread #59308 confirms it surprises researchers. Consensus-level finding — no empirical study needed.

**Finding 4: The GitHub Search API caps results at 1,000 — systematic research requires multi-query decomposition**
- Claim: GitHub's Search API returns at most 1,000 results per query regardless of actual match count. For any research topic with >1,000 relevant repositories, a single query produces a biased sample (typically top-ranked by stars or relevance, not comprehensive). The documented workaround is multi-query decomposition: iterate over date ranges, star-count bands, or language filters to partition the result space into segments each under 1,000, then union results. Without this, systematic retrieval is structurally impossible using GitHub search alone.
- Source: GitHub ossf/criticality_score issue #33; PyGithub issue #824; PyGithub issue #1072; GitHub Community Discussion #64629
- Confidence: CONSENSUS
- Verify note: Hard API constraint documented by GitHub. The multi-query decomposition workaround is confirmed by multiple independent engineering teams working around the same limit.

**Finding 5: Awesome lists function as curated research indexes, not endpoints — the meta-list (sindresorhus/awesome) is the entry point**
- Claim: The awesome-list ecosystem (anchored by sindresorhus/awesome, which has its own `awesome-list` and `curated-list` GitHub topics) provides pre-screened domain-specific indexes maintained by domain practitioners. For a research topic, finding the domain's awesome list gives a vetted starting set that bypasses star inflation and topic-tag fragmentation. The discovery path is: search `topic:awesome-list [domain]` or `awesome [domain] in:name` → use the list as an index, not as an endpoint → follow listed repositories to primary sources. The failure mode: using awesome lists as quality validators rather than discovery tools; their inclusion criteria vary widely.
- Source: github.com/sindresorhus/awesome (138k+ stars); github.com/topics/awesome-list; github.com/topics/awesome
- Confidence: CONSENSUS
- Verify note: Well-established practitioner pattern. The distinction between using awesome lists as indexes vs. endpoints is a practitioner consensus claim, not empirical — included at consensus confidence.

**Finding 6: Paper-to-repo and repo-to-paper bidirectional links are a structural discovery path — Papers With Code is the primary index**
- Claim: A study of 20,000 GitHub repositories referencing academic papers (arxiv:2004.00199, Wattanakriengkrai et al., 2020) found that the most-referenced arXiv papers are in ML/deep learning/computer vision, and that papers with linked code see ~20% higher monthly citation rates after the repository is created (Information Processing and Management, 2023). The bidirectional discovery path: (1) arXiv paper → find linked GitHub repo (via Papers With Code, or via arXiv abstract's "code" section, or via the paper's References section); (2) GitHub repo → find linked papers (search `in:readme arxiv.org` or `in:readme doi.org`). Papers With Code (paperswithcode.com) is the primary structured index for this path in ML/AI research — it links ~100k+ papers to implementations.
- Source: arxiv:2004.00199 (Wattanakriengkrai et al., 2020); doi:10.1016/j.ipm.2023.103477 (papers with/without code citation study); paperswithcode.com; arxiv:2004.06059 (paper2repo, 2020)
- Confidence: HIGH (bidirectional link structure); MEDIUM (20% citation rate figure — single study)
- Verify note: The bidirectional structure is well-documented. The 20% citation advantage is from a single study (2023) — medium confidence. Papers With Code as index is practitioner consensus + documented.

## Practical implications for kit procedures

**Rule 1 — Multi-strategy minimum (not topic-only):**
Any GitHub discovery pass must execute at minimum three complementary strategies: (a) topic search `topic:[term] topic:[term-plural]`, (b) README search `[term] in:readme`, and (c) awesome-list discovery `awesome [domain] in:name`. Topic-only search structurally misses the majority of relevant repositories due to low adoption. A procedure that runs only topic search without README and awesome-list passes has known structural recall failures.

**Rule 2 — Quality floor for claim extraction:**
Apply `stars:>=500 pushed:>2025-01-01 archived:false` as the minimum quality gate before extracting claims from a repository. Below 500 stars, fake-star contamination and abandonment rates make the quality signal unreliable. Check fork:star ratio for repos above 5k stars — ratio below 50/1k is a fake-star indicator.

**Rule 3 — Singular AND plural topic search:**
Always search both `topic:[term]` and `topic:[terms]` (plural variant). GitHub topic matching is exact — `topic:llm-agent` and `topic:llm-agents` return different non-overlapping sets. Missing either cuts recall for the affected term.

**Rule 4 — Paper-first path for ML/AI research:**
When researching ML/AI topics, use Papers With Code as a primary discovery entry point before GitHub search. The paper-to-repo link is more reliable than topic-tag search because arXiv authors explicitly link their implementations. The discovery path: `paperswithcode.com/sota/[task]` → linked repositories → related repositories in the implementation's README.

**Rule 5 — 1,000-result cap workaround:**
If a topic has >1,000 relevant repositories (likely for broad ML/AI topics), use date-range decomposition: split queries by `pushed:[date-range]` bands, or by `stars:[range]` bands, to keep each query under 1,000 results. Union results by repository URL to deduplicate. A single query against a broad topic produces a top-1,000 popularity-ranked sample, not a comprehensive set.

**Rule 6 — Awesome-list as index, not validator:**
Use awesome lists to discover repositories, not to certify their quality. Inclusion criteria vary by maintainer. After finding repositories via an awesome list, apply the same quality gate (Rule 2) before extracting claims.

**Intake question:** When a research angle targets a GitHub discovery pass, ask: is this a narrow named tool/technique (topic-search sufficient) or a broad domain (multi-strategy required, 1,000-cap workaround needed)?

**Adversarial query result:**

Found: GitHub API caps at 1,000 results per search (hard limit, no override). GitHub does not expose all events (pushes, releases, clone operations). Counter-case constructed: A researcher who follows the procedure rules above still faces irreducible recall gaps: (a) private/unlisted repositories are invisible regardless of strategy; (b) repositories that never gained traction (no stars, no README) represent genuine work that is structurally undiscoverable; (c) topic-tag search and awesome-list search both have coverage limited to repositories whose maintainers chose to participate in the tagging/curation ecosystem; (d) Papers With Code focuses on ML/AI — other domains have weaker linking infrastructure. The practical implication: GitHub broad discovery produces a sample biased toward popular, ML-adjacent, English-language, maintainer-engaged projects. This is a known limitation, not a correctable procedure failure.

## Key sources

1. **Wermke et al. (2024/2026) — "Six Million (Suspected) Fake Stars in GitHub"** (arXiv:2412.13459, ICSE 2026, CMU/NCSU/Socket). 20 TB of GitHub metadata analyzed; 6M+ fake stars detected; 16.66% of repos with 50+ stars affected by mid-2024. Primary source for fake-star quality-signal degradation.

2. **Wattanakriengkrai et al. (2020) — "GitHub Repositories with Links to Academic Papers: Public Access, Traceability, and Evolution"** (arXiv:2004.00199, ScienceDirect doi:10.1016/j.ipm.2023.103477). 20,000 repositories studied; bidirectional paper-repo linking structure; papers with linked code see ~20% higher citation rates. Primary source for paper-to-repo discovery path.

3. **Garousi, Felderer, Mäntylä (2019) — "Guidelines for including grey literature and conducting multivocal literature reviews in software engineering"** (arXiv:1707.02553, ScienceDirect). Establishes MLR as the framework for incorporating GitHub as grey literature in systematic reviews. Foundational source for treating GitHub discovery as a required complement to academic search.

4. **GitHub Blog — "Topic Suggestions for Millions of Repositories"** (github.blog/engineering/user-experience/topics/). GitHub's own documentation of the low-adoption problem that motivated their ML-based topic suggestion system. Confirms that most repositories do not self-tag with topics.

5. **Zampetti et al. (2022) — "Automatically Categorising GitHub Repositories by Application Domain"** (arXiv:2208.00269). Builds an automated domain classifier for GitHub repos; confirms that existing topic-tag-based categorization is insufficient and requires ML augmentation. Supports the low-adoption finding.
