## 1. Semantic Scholar API vs Semantic Scholar Web

### What the API provides that the web interface does not

**Structured output.** The web interface returns HTML for human reading. The API returns JSON with every field addressable: title, authors, abstract, AI TLDR, citation count, influential citation count, reference count, publication date, venue, journal, DOI, arXiv ID, PubMed ID, fields of study, publication types, open-access PDF link, SPECTER2 embeddings. No scraping or parsing HTML.

**Bulk search.** Endpoint: `GET /graph/v1/paper/search/bulk`. Returns up to 10M matching papers total, paginated at 1,000 per response. The regular relevance search endpoint (`/paper/search`) is heavier and slower; bulk is the recommended path for any volume task.

**Citation graph traversal.** Endpoints:
- `GET /graph/v1/paper/{paper_id}/citations` — all papers that cite a given paper
- `GET /graph/v1/paper/{paper_id}/references` — all papers that a given paper cites

The web interface shows these in a browsable list. The API returns them as structured data you can iterate, filter, and pipeline into further lookups. The full citation graph at snapshot scale is in the Datasets API (see below).

**Batch paper details.** `POST /graph/v1/paper/batch` — submit multiple paper IDs in one call, get all their fields back at once. The web has no equivalent.

**Author batch endpoint.** `GET /graph/v1/author/batch` — fetch name, h-index, paper count, full paper list for multiple author IDs in one call.

**SPECTER2 embeddings.** Papers include pre-computed dense vector embeddings. Enables semantic similarity lookups and clustering without a separate embedding step. Not accessible via web.

**Datasets API.** Base URL: `https://api.semanticscholar.org/datasets/v1`. Provides monthly full-corpus snapshots and incremental diffs between releases. Includes citation edges with citation context and intent (supporting/contrasting/etc.) and influential classification. **This endpoint requires an API key.**

**Customizable field projection.** Every request accepts a `fields` query parameter listing exactly which fields to return. Reduces payload and latency.

### Access requirements and rate limits

| Access type | Rate limit | Notes |
|---|---|---|
| No API key | ~5,000 req / 5 min **shared** across all unauthenticated users | Effectively unpredictable under load |
| API key (free, self-serve) | 1 RPS across all endpoints | Apply at semanticscholar.org/product/api |
| Higher rate | Available after review | Request via the API portal |
| Datasets API | API key required | Also required for incremental diff endpoint |

API key signup is free, self-serve, and takes under a minute. There is no paid tier listed; higher rates are granted on request based on stated use case.

### When API is worth the setup cost vs just using the web

**Use API when any of these are true:**
- Retrieving more than 20–30 papers (manual web pagination is immediately slower than one bulk call)
- Need citation counts, reference lists, or who-cites-this as data (not as a webpage to read)
- Running any automated pipeline (nightly updates, batch enrichment, building a corpus)
- Need SPECTER2 embeddings for semantic grouping
- Need to follow citation chains two or more hops deep
- Downloading full dataset snapshots for local indexing

**Use web when:**
- Quick relevance check on a single paper or author
- One-off lookup where reading the result is the entire task
- No code environment available

## 2. arXiv API vs arXiv Web Search

### Endpoint structure

Base: `http://export.arxiv.org/api/query`

Primary parameters:

| Parameter | Function |
|---|---|
| `search_query` | Free-text query with field prefixes and boolean operators |
| `id_list` | Comma-delimited arXiv IDs for direct lookup |
| `start` | Pagination offset (0-indexed) |
| `max_results` | Results per request; max 2,000; total retrievable per query: 30,000 |
| `sortBy` | `relevance`, `lastUpdatedDate`, `submittedDate` |
| `sortOrder` | `ascending`, `descending` |

### Field prefixes for `search_query`

| Prefix | Searches in |
|---|---|
| `ti:` | Title |
| `au:` | Author |
| `abs:` | Abstract |
| `co:` | Comment field |
| `jr:` | Journal reference |
| `cat:` | Subject category (e.g. `cat:cs.LG`) |
| `all:` | All fields simultaneously |

Boolean operators: `AND`, `OR`, `ANDNOT`. Parentheses for grouping.

Example: `ti:transformer AND au:vaswani` — papers with "transformer" in title authored by Vaswani.

### Date filtering

`submittedDate:[YYYYMMDDTTTT+TO+YYYYMMDDTTTT]` in GMT (24h format). Filters by submission date, not publication date.

### Output format

Atom 1.0 XML. Machine-readable without scraping. Contains title, authors, abstract, arXiv ID, primary category, all categories, submission date, last update, DOI (if assigned), journal ref, comment, links to abstract page and PDF.

### Rate limits and access

No authentication required. No hard daily cap documented. arXiv asks for a 3-second delay between successive requests. For truly bulk access (full corpus download), arXiv provides a separate bulk data endpoint; the API is not designed for full-corpus harvesting.

### What the API does better than the web

**Field-precise boolean queries.** Web search has basic keyword search with limited operator support. API allows exact construction: `(ti:attention OR ti:transformer) AND cat:cs.AI AND au:vaswani`. Web cannot match this precision.

**Date-range filtering.** Web browsing by date is manual. API `submittedDate` filter gives exact windows programmatically.

**Programmatic retrieval up to 30,000 results.** Web paginates at ~25 results with manual click-through. API returns 2,000 per call.

**Machine-readable output.** Atom XML is directly parseable. Web output requires HTML parsing or scraping.

**Direct ID lookup.** `id_list=2106.09685` returns that exact paper's metadata in one call with no HTML to parse.

**Category browsing as a query dimension.** `cat:cs.CL` precisely scopes to Computation and Language. Web category browsing exists but is not combinable with other query operators.

## 3. Exa.ai as Grey-Literature Search

### What Exa is

Exa is a search engine designed for AI agents that uses neural embeddings to rank results by semantic similarity to a query, rather than keyword frequency or PageRank-style link counting. The distinction: Google matches words; Exa matches meaning. A query like "companies building RAG for legal documents" returns conceptually matched results with Exa; with Google it returns pages that contain those exact words together.

### Search types and latency

| Mode | Latency | Best for |
|---|---|---|
| Instant | ~200ms | Latency-critical, lower relevance ceiling |
| Fast | ~450ms | Speed with minimal quality loss |
| Auto (default) | ~1s | General-purpose; combines neural + keyword |
| Deep-lite | ~2–10s | Light synthesis tasks |
| Deep | ~5–60s | Complex multi-source research |
| Deep-reasoning | ~10–60s | Multi-step reasoning queries |

Neural mode and keyword mode are separately selectable. In benchmark tests, neural mode returned semantically relevant results where keyword mode returned unrelated businesses matching the same terms literally.

### Operator support

Exa supports a structured parameter set, not a query syntax for inline operators:

| Parameter | Function |
|---|---|
| `includeDomains` | Whitelist: results only from these domains |
| `excludeDomains` | Blacklist: no results from these domains |
| `startPublishedDate` / `endPublishedDate` | Filter by publication date (ISO 8601) |
| `startCrawlDate` / `endCrawlDate` | Filter by date Exa indexed the page (ISO 8601) |
| `category` | Scoped index: `research_paper`, `news`, `company`, `personal_site`, etc. |
| `includeText` / `excludeText` | Require or exclude specific text in results |
| `numResults` | Base count; first 10 included in base price |

The `research_paper` category supports all filters including domain and date. The `company` and `people` categories have restricted filter support.

### Coverage for CS/AI grey literature

Exa's index includes 100M+ research papers, plus general web. The `research_paper` category biases retrieval toward scholarly content. For grey literature — practitioner blog posts, technical documentation, ML engineering writeups, project pages — Exa's neural mode surfaces conceptually matched content that standard keyword search misses when terminology varies.

### Pricing and access

- **Free tier:** 20,000 requests/month with full API access
- **Standard search (pay-as-you-go):** $7 per 1,000 requests (up to 10 results each)
- **Deep search:** $12 per 1,000 requests
- **Deep-reasoning:** $15 per 1,000 requests
- **Contents API:** $1 per 1,000 pages (to fetch full text of results)
- Additional results beyond 10: $1 per 1,000 requests

API key required; free signup.

### Verdict: when Exa is meaningfully better than Google for our use case

**Exa wins clearly when:**
- Query intent is conceptual rather than terminological ("ML papers about training instability in large models" — Exa handles vocabulary variation; Google requires the user to enumerate synonyms)
- Searching practitioner content where exact title/keyword is unknown (e.g., "blog posts about deployment failure modes for RAG systems")
- Domain-scoped retrieval with semantic matching (combine `includeDomains: [huggingface.co, bair.berkeley.edu]` with a concept query)
- Finding content that uses different vocabulary than the query (cross-disciplinary concepts, new terminology, code-adjacent writing)

**Google wins or ties when:**
- You know the exact title, author, or quoted string
- You need `site:` scoping for a single known domain (free vs. $7/1000 for Exa)
- Broad coverage matters more than precision (Google's index is larger)
- The query is already a precise keyword match (no benefit from neural ranking)

**Bottom line:** Exa is not a Google replacement. It is a complement for queries that are concept-shaped rather than keyword-shaped, and where grey literature coverage matters more than exhaustive web coverage. The 20,000 free requests/month is sufficient for research workloads that are not production-scale.

## 4. Decision Matrix

| Research intent | Tool | Rationale |
|---|---|---|
| Find 100+ papers on a narrow topic | **Semantic Scholar API** (`/paper/search/bulk`) | Structured JSON, up to 10M results, field filtering, SPECTER2 for semantic match, free with API key; arXiv API is the fallback if preprint-only scope is sufficient |
| Quick relevance check on 3 terms | **Semantic Scholar web** or **arXiv web** | No setup, immediate interactive feedback, reading the result is the whole task |
| Find practitioner blog posts and grey lit | **Exa.ai** (neural mode, auto category) | Concept-matched retrieval, domain filtering, date filtering; Google as fallback for known-exact queries |
| Find active GitHub repos | **GitHub native search** (web or API) | `topic:X language:python stars:>100 archived:false created:>2024-01-01` — these qualifiers are native to GitHub; no other tool matches GitHub's own index for repo-level filtering; Exa neural mode as supplement for concept-based repo discovery |
| Find papers that cite a specific paper | **Semantic Scholar API** (`/paper/{id}/citations`) | Only tool that returns the complete structured citation list as data; web shows the same but as HTML requiring manual reading; OpenCitations/COCI is an alternative with 1B+ citation links if Semantic Scholar coverage is insufficient for a given paper |

## Notes on tool gaps not covered above

**OpenAlex** (not researched in depth here) covers 240M+ works with a free API, no rate limit requiring key, and topic hierarchies. It is a strong alternative to Semantic Scholar for bulk retrieval and may have broader non-CS coverage. Not evaluated as a primary tool for this kit but should be on the radar for wave 2.

**Google Scholar** has no public API and prohibits scraping. Not usable programmatically.

**PubMed/NCBI E-utilities** — relevant if scope extends to biomedical AI. Not in scope for this angle.
