## 1. Field Qualifiers — Complete Reference

### API-Level Field Prefixes (used in export.arxiv.org/api/query and direct search URLs)

| Prefix | Field Searched | Notes |
|--------|---------------|-------|
| `ti:` | Title only | Most precise signal for what a paper is about; authors choose titles deliberately |
| `abs:` | Abstract only | Covers methodology + keywords in depth; best for concept coverage |
| `au:` | Author name | Recommended format: `surname, forename` or `surname, initial`; quoted form returns exact match only |
| `all:` | All fields combined | Title + abstract + author + comments + report number; broadest net |
| `cat:` | Subject category | Exact category code (`cat:cs.AI`) or wildcard (`cat:cs*`); scopes to announced category |
| `co:` | Comments field | Author-added notes (page counts, conference info); rarely useful for topic search |
| `jr:` | Journal reference | Published venue; useful for known conference proceedings (e.g., NeurIPS) |
| `rn:` | Report number | Institutional report IDs; specialist use only |
| `submittedDate:` | Submission date | Range syntax: `[YYYYMMDDTTTT TO YYYYMMDDTTTT]` (24-hour GMT); partial dates valid |

**Claim A1:** The `id:` prefix (e.g., `id:2301.00234`) exists in documentation but the official API manual states to use the `id_list` parameter instead for version-safe retrieval. Do not rely on `search_query=id:xxx` for precise paper lookup.

**Claim A2:** The `all:` qualifier is the default when no prefix is specified in the web UI. In the API it must be specified explicitly.

**When to prefer each:**
- Use `ti:` when the term is likely in the title and you want high-precision results (fewer false positives).
- Use `abs:` when the concept may not appear in the title but is central to the paper's method or evaluation.
- Use `ti:` AND `abs:` separately with OR to catch papers where the term appears in either without requiring both.
- Use `all:` only when a term is obscure enough that its appearance anywhere is a positive signal.
- Use `cat:` as a filter, not a primary search axis — it scopes results to a community, not a topic.
- Use `au:` for known researcher tracking, not topic discovery.

## 2. Boolean Operators

### Supported Operators

| Operator | Function | Precedence |
|----------|----------|-----------|
| `AND` | Both conditions required | Standard conjunction; evaluated before OR without parentheses |
| `OR` | Either condition sufficient | Lower precedence than AND |
| `ANDNOT` | Excludes right-hand condition | Subtraction; evaluated after AND |

**Claim B1:** The arXiv API documentation does NOT list a standalone `NOT` operator. The correct negation form is `ANDNOT`. Using `NOT` alone produces undefined behavior.

**Claim B2:** Operator precedence is NOT explicitly documented with a formal grammar in the arXiv API manual. In practice, AND binds more tightly than OR, consistent with standard Boolean algebra. Use parentheses to enforce intended evaluation order.

### Grouping with Parentheses

Parentheses must be URL-encoded in API query strings:
- `(` → `%28`
- `)` → `%29`

Example (raw): `au:del_maestro ANDNOT (ti:checkerboard OR ti:Pyrochlore)`
Example (URL-encoded): `au:del_maestro+ANDNOT+%28ti:checkerboard+OR+ti:Pyrochlore%29`

**Claim B3:** In the web UI advanced search form, parentheses can be used directly in the search box — URL encoding is only required when constructing raw API GET requests.

### Phrase Matching

Wrap multi-word phrases in double quotes for exact sequence matching:
- Raw: `ti:"task decomposition"`
- URL-encoded: `ti:%22task+decomposition%22`

Quoted phrases match only that exact token sequence. Unquoted terms are treated as independent tokens that may appear non-adjacently.

## 3. Category Codes for CS/AI

### Primary Categories for AI/ML/Agent Research

| Code | Full Name | Scope Relevant to Kit-Maker Topics |
|------|-----------|-------------------------------------|
| `cs.AI` | Artificial Intelligence | Planning, knowledge representation, expert systems, classical AI; NOT the main venue for LLM work |
| `cs.LG` | Machine Learning | Dominant venue for LLM training, RL, fine-tuning, evaluation methodology; highest volume |
| `cs.CL` | Computation and Language | NLP, LLMs as language systems, instruction following, text generation; overlaps heavily with cs.LG |
| `cs.MA` | Multiagent Systems | Multi-agent coordination, distributed AI, agent communication protocols |
| `cs.SE` | Software Engineering | Code generation, testing, debugging tools, software agents, program synthesis |
| `cs.IR` | Information Retrieval | RAG, search agents, document retrieval, query formulation |
| `cs.HC` | Human-Computer Interaction | User-facing agent systems, human-AI teaming |
| `cs.RO` | Robotics | Embodied agents, physical task planning |
| `cs.NE` | Neural and Evolutionary Computing | Neural architectures underlying agents; less relevant for behavioral/planning topics |
| `cs.CV` | Computer Vision | Vision-language models, multimodal agents |

**Claim C1:** For LLM agent research as of 2025-2026, `cs.LG` and `cs.CL` have the highest submission volume. Many LLM agent papers are cross-listed across `cs.AI`, `cs.LG`, and `cs.CL` simultaneously.

**Claim C2:** `cs.AI` carries a misleading name from an era before deep learning. Modern neural/LLM work rarely lists `cs.AI` as primary; it is primarily classical AI (search, theorem proving, planning formalisms). Filtering only to `cs.AI` will miss most LLM agent papers.

**Claim C3:** Category assignments are author-supplied and incomplete. A paper about "LLM behavioral compliance" could be submitted under cs.CL, cs.LG, cs.AI, or cs.SE depending on framing. Category-scoped queries should supplement, not replace, keyword queries.

### When to Scope by Category vs. Search All Fields

**Scope by category when:**
- You want only community-vetted work in a discipline (e.g., `cat:cs.MA` for formal multi-agent theory)
- You want to exclude adjacent but irrelevant communities (e.g., prevent biology or physics cross-list noise)
- You already have a good keyword query and want to narrow it

**Search all fields when:**
- The topic is interdisciplinary and community boundaries don't align with the concept
- The research topic is new enough that category norms haven't stabilized
- You are doing discovery, not confirmation

## 4. Practical Patterns for Kit-Maker Topics

### 4.1 Topic: "LLM Agent Planning"

**Naive query:**
```
all:LLM agent planning
```
Problems: "LLM", "agent", "planning" are each common words that appear in unrelated papers. Returns thousands of loosely related results. "LLM" is an acronym that may not appear consistently (some papers use "large language model" spelled out).

**Improved query:**
```
(ti:"LLM agent" OR ti:"language model agent" OR ti:"LLM-based agent") AND (abs:planning OR abs:"task planning" OR abs:"action planning")
```
Why better: Anchors the agent framing in the title (deliberate author signal) while allowing planning vocabulary to vary in the abstract. Handles both "LLM" abbreviation and spelled-out form.

**Further refinement with category:**
```
(ti:"LLM agent" OR ti:"language model agent") AND (abs:planning OR abs:decomposition) AND (cat:cs.AI OR cat:cs.LG OR cat:cs.CL)
```

---

### 4.2 Topic: "Behavioral Compliance in LLMs"

**Naive query:**
```
all:"behavioral compliance" LLM
```
Problem: "behavioral compliance" is not an established term of art in ML; papers use "instruction following", "rule adherence", "constraint satisfaction", "alignment", "safety". Returns near-zero results.

**Improved query:**
```
(ti:"instruction following" OR ti:"instruction adherence" OR ti:"constraint following" OR ti:"rule following") AND (abs:LLM OR abs:"language model") AND cat:cs.CL
```
Why better: Matches the actual vocabulary used in the literature. Category scoping to `cs.CL` removes robotics and formal verification noise.

**Alternate formulation (broader):**
```
abs:"instruction following" AND abs:"large language model" AND (abs:compliance OR abs:adherence OR abs:violation OR abs:audit)
```

---

### 4.3 Topic: "Multi-Step Task Decomposition"

**Naive query:**
```
all:"task decomposition" agent
```
Problem: "task decomposition" is used in robotics, formal planning, HCI, and project management — all outside the LLM agent scope.

**Improved query:**
```
(ti:"task decomposition" OR ti:"chain of thought" OR ti:"plan decomposition" OR ti:"subgoal") AND (abs:"language model" OR abs:LLM) AND (cat:cs.AI OR cat:cs.LG OR cat:cs.CL)
```
Why better: Covers synonym families ("chain of thought" is the dominant framing for step-by-step reasoning in LLM papers), anchors in language model context, and scopes to relevant communities.

**Note on term evolution:** "Chain of thought" entered as the dominant phrasing circa 2022; "task decomposition" is more common in pre-2022 work and in robotics. For cross-era coverage, include both.

## 5. Known Failure Modes

### FM-1: Vocabulary Mismatch (Under-retrieval)
The arXiv full-text search covers only title, abstract, and comments — not body text. If the concept you're searching for is named differently by different subcommunities, a single-term query misses entire research threads.

Example: "behavioral compliance" → misses papers using "alignment", "instruction following", "safety constraints", "policy adherence".

Mitigation: Build OR-expanded synonym clusters for every key concept before querying.

### FM-2: Acronym Inconsistency (Under-retrieval)
Authors inconsistently use acronyms vs. spelled-out forms. "LLM" vs. "large language model" vs. "GPT-based" vs. "foundation model" refer to overlapping but non-identical sets.

Mitigation: Always OR the abbreviation with the spelled-out form for primary concepts: `(ti:LLM OR ti:"large language model")`.

### FM-3: Category Over-Scoping (Under-retrieval)
Adding `cat:cs.AI` expecting to catch "AI papers" actually narrows to classical AI work and misses the vast majority of LLM research, which lives in `cs.LG` and `cs.CL`.

Mitigation: When uncertain, use OR across relevant categories or drop the category filter entirely for discovery queries.

### FM-4: Over-Quoted Phrases (Under-retrieval)
Quoting a long phrase like `"multi-step reasoning agent planning pipeline"` returns only papers with that exact string. Authors rarely use identical phrasing.

Mitigation: Quote 2-3 word anchors only. Prefer `"task decomposition"` over `"multi-step task decomposition agent"`.

### FM-5: AND-Heavy Queries (Under-retrieval)
Chaining too many AND conditions creates a query that no paper satisfies unless it was written to match your exact framing. Each AND constraint reduces the candidate set multiplicatively.

Mitigation: Move secondary concepts to abs: with OR instead of requiring all terms. Use a tiered approach: first run with core 2-3 terms, then narrow with an additional AND only if results are unmanageably large (>500).

### FM-6: Author Name Ambiguity
Author searches without quoted exact names match partial names. `au:smith` matches every Smith, Smithson, Blacksmith, etc.

Mitigation: Always quote author names: `au:"smith, j"` or `au:"smith, john"`.

### FM-7: Category Drift Over Time (Wrong Papers)
arXiv categories are self-reported and community norms shift. A category that was meaningful in 2018 may have been diluted by 2025. `cat:cs.AI` is a documented example of this drift.

Mitigation: Verify category scoping assumptions by sampling 10-20 recent papers before relying on a category filter.

### FM-8: Missing Cross-Lists (Under-retrieval)
A paper may be submitted under `cs.LG` as primary but cross-listed under `cs.AI`. A `cat:cs.AI` filter retrieves it; a `cat:cs.LG` filter also retrieves it. But if you use a category filter that excludes one, you may miss the primary-category submission.

Mitigation: OR the category filter across all relevant codes rather than picking one.

## 6. Cross-Source Translation Notes

### arXiv → Semantic Scholar
Semantic Scholar does NOT support boolean operators (AND/OR/NOT) or wildcards in its web UI. It uses semantic/embedding-based ranking, not keyword boolean logic.

Translation strategy:
- Convert arXiv boolean OR-clusters to individual separate queries, run each, then merge results manually.
- Use quoted phrases (the only supported advanced syntax) for high-precision anchors.
- Semantic Scholar's API (api.semanticscholar.org) DOES support boolean logic as a filter parameter — use programmatic access for complex queries.
- Semantic Scholar covers the same arXiv papers plus additional published venues (ACL Anthology, NeurIPS proceedings, etc.) — it catches peer-reviewed versions that arXiv preprints may not link to.

### arXiv → Google Scholar
Google Scholar has limited and poorly documented boolean support. Key differences:
- `AND` is implicit between unquoted terms (same as arXiv default).
- `OR` is supported in uppercase.
- No field prefix system equivalent to `ti:` or `abs:` — the closest is `intitle:` for title-scoped terms.
- No programmatic API.
- Coverage includes non-arXiv papers; useful for finding published conference proceedings that authors never posted to arXiv.

Translation pattern for `ti:"task decomposition" AND abs:"language model"`:
Google Scholar equivalent: `intitle:"task decomposition" "language model"` (abstract scoping not available; terms outside intitle: match anywhere in the indexed record).

### arXiv → ACL Anthology (for NLP/CL-adjacent topics)
ACL Anthology (aclanthology.org) covers papers from ACL, EMNLP, NAACL, AAAI NLP tracks — often the peer-reviewed versions of arXiv cs.CL preprints.
- Boolean support: AND/OR through its Lucene-based search.
- Field scoping: title:, author:, year:.
- Venue filter available (restrict to specific conference).
- For NLP-heavy kit-maker topics (instruction following, behavioral compliance), ACL Anthology often surfaces higher-quality curated work vs. arXiv's preprint volume.

## 7. Quick Reference: Query Construction Checklist

1. Identify 2-3 core concepts for the topic.
2. For each concept, list 2-4 synonyms/alternate phrasings used in the literature.
3. Build an OR cluster for each concept.
4. Combine clusters with AND.
5. Decide on field scoping: title for high precision, abstract for recall, all for discovery.
6. Add category filter only to exclude clearly irrelevant communities.
7. Test: if results > 500, add one more AND; if < 20, remove one AND or broaden an OR cluster.
8. Check: did you cover both abbreviation and spelled-out forms for all acronyms?
