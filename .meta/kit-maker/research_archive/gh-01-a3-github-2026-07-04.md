## 1. Repository Search Operators — Exact Syntax

All operators work in GitHub's search bar at github.com/search and via `gh search repos` CLI.
Operators combine with AND by default; use `OR` explicitly for alternatives.

### Scoping operators (where to look)

| Operator | What it does | Example |
|---|---|---|
| `in:name` | Match keyword in repository name only | `agent in:name` |
| `in:description` | Match keyword in repo description | `llm in:description` |
| `in:readme` | Match keyword in README content | `benchmark in:readme` |
| `in:topics` | Match keyword in topic tags | `agent in:topics` |
| `in:name,description` | Match in name OR description (comma = OR within in:) | `scaffold in:name,description` |

When `in:` is omitted, GitHub searches name, description, and topics — NOT README content. README content requires explicit `in:readme`.

### Filtering operators

| Operator | Syntax | Example | Notes |
|---|---|---|---|
| `language:` | `language:LANGUAGE` | `language:python` | Case insensitive; use display name (Python, JavaScript, Rust) |
| `stars:` | `stars:n`, `stars:>=n`, `stars:n..n` | `stars:>=500`, `stars:100..5000` | Range via `..` notation |
| `forks:` | `forks:n`, `forks:>=n`, `forks:<n`, `forks:n..n` | `forks:>=50` | Same range syntax as stars |
| `pushed:` | `pushed:>YYYY-MM-DD`, `pushed:>=YYYY-MM-DD` | `pushed:>2025-01-01` | ISO 8601; filters by last commit push date, not merge date |
| `created:` | `created:<YYYY-MM-DD`, `created:>YYYY-MM-DD` | `created:>2023-01-01` | Repository creation date |
| `topic:` | `topic:TOPIC` | `topic:llm-agent` | Exact tag match only; see section 2 |
| `license:` | `license:LICENSE_KEYWORD` | `license:mit`, `license:apache-2.0` | Use exact SPDX key: apache-2.0, mit, gpl-3.0, cc-by-4.0 |
| `size:` | `size:n`, `size:>=n`, `size:n..n` | `size:50..5000` | In kilobytes; useful to exclude tiny scaffolds or massive data repos |
| `archived:` | `archived:true` or `archived:false` | `archived:false` | Default search includes archived repos — always add this for research |
| `mirror:` | `mirror:true` or `mirror:false` | `mirror:false` | Excludes mirrors of external repos (e.g., GitHub mirrors of GitLab projects) |
| `is:` | `is:public` or `is:private` | `is:public` | Visibility filter |
| `org:` | `org:ORGNAME` | `org:microsoft` | Restrict to one organization |
| `user:` | `user:USERNAME` | `user:karpathy` | Restrict to one user's repos |
| `NOT` | `NOT term` | `NOT awesome in:name` | Exclude term; useful to filter out curated lists |

### Compound example

```
topic:llm-agent language:python stars:>=500 pushed:>2025-01-01 archived:false
```

Finds Python LLM-agent repos, 500+ stars, pushed since Jan 2025, not archived.
As of July 2026: approximately 150-400 results depending on exact date cutoff.

## 2. Topic Search — How GitHub Topics Work

### Mechanics

Topics are free-text tags that repo owners add manually via "Manage topics" on the repo landing page. Any string is valid. GitHub maintains a curated set of "featured" topics (e.g., `machine-learning`, `deep-learning`) but enforcement is absent — anyone can tag anything.

The `topic:TAG` operator matches only. It does not use partial matching or synonyms.

### Discovery qualifiers for topic pages themselves

These filter topic pages at github.com/topics (not repos directly):

| Qualifier | What it does |
|---|---|
| `is:featured` | Topics with the most repositories on GitHub (GitHub's curated set) |
| `is:curated` | Topics with community-contributed descriptions, images, and links |
| `repositories:>=n` | Topics with at least n repositories |

### Relevant topic clusters for AI/CS research (July 2026 state)

**Agent architectures:**
- `topic:llm-agent` — 1,630 repos; top: autogen (59k stars), agentscope (27k), letta (23k)
- `topic:llm-agents` — variant spelling; significant overlap with above; search both
- `topic:autonomous-agents` — broader; includes non-LLM automation; sorted by `?o=desc&s=updated`
- `topic:ai-agents` — widest net; high noise
- `topic:agentic-framework` — more specific to scaffolding frameworks; lower volume, higher signal

**Behavior / enforcement:**
- `topic:llm-guardrails` — output validation, safety enforcement
- `topic:prompt-engineering` — high volume, mixed signal (tutorials outnumber implementations)

**Research artifacts:**
- `topic:research-papers` — curated paper lists and paper-linked implementations
- `topic:benchmark` — evaluation suites

### Topic search limitations

- Self-assigned: no enforcement. A repo tagged `topic:llm-agent` may be tangentially related.
- Popular topics accumulate noise faster than narrow ones. `topic:ai-agents` is noisier than `topic:agentic-framework`.
- Exact match only: `topic:llm-agent` does not return `topic:llm-agents`. Search both spellings for common variant pairs.
- New repos frequently have no topics at all. Use `in:readme` to catch what topics miss.
- Topics are not hierarchical: there is no parent-child relationship between `topic:agents` and `topic:llm-agents`.

## 3. Quality Signals for Repository Relevance

### 3.1 Star count tiers

Stars are noisy (see failure modes) but the fastest first-pass filter.

| Tier | Star range | Interpretation |
|---|---|---|
| Noise floor | 0–50 | Experiment, personal project, or very early-stage. Ignore unless linked from a paper or by a known researcher. |
| Emerging | 50–500 | Community awareness forming. Worth examining if topic is narrow or if the repo has paper backing. |
| Established | 500–2,000 | Active community. Dependency graph starting to form. Claims likely reality-tested. |
| Significant | 2,000–10,000 | Production-adjacent. Likely cited in papers, has issues from real-world users, extractable claims. |
| Flagship | 10,000+ | Enterprise-grade or viral. Benchmark papers, extensive docs, contributor diversity. |

For research claim extraction: the practical minimum is approximately 500 stars paired with a recent push. Below 500, claims are provisional. Above 2,000, claims have likely been battle-tested by external users.

### 3.2 Commit recency — pushed: thresholds

| Window | Interpretation |
|---|---|
| `pushed:>2025-07-01` (within ~12 months) | Actively maintained. Treat as current. |
| `pushed:>2024-01-01` (12–24 months ago) | Dormant but not dead. APIs may still be valid; check issues for deprecation notices. |
| `pushed:>2022-01-01` (2–4 years ago) | Maintenance mode or effectively abandoned. Claims require independent verification against current landscape. |
| Older than 4 years | Historical artifact. Claims likely superseded. Use only as provenance for tracing methodology origin. |

For AI research specifically: the landscape moves faster than general software. An LLM agent framework last pushed in 2023 is likely obsolete for implementation purposes but may still contain architectural claims worth extracting.

Default threshold for active-research discovery: `pushed:>2025-01-01`.

### 3.3 Issue activity — production use vs. experiment

**Production-use signals:**
- Bug reports referencing specific model versions, API providers, or integration contexts (e.g., "fails with Claude 3.5 Sonnet tool_use")
- Feature requests that imply deployment at scale ("rate limiting", "multi-tenant", "enterprise auth", "cost optimization")
- Issues filed by accounts with real contribution history, not throwaway accounts
- Presence of "good first issue" and "help wanted" labels — indicates maintainers are actively triaging and want contributors
- Closed-to-open ratio above 60%: maintainers are responsive
- Average time-to-first-response under 7 days on recent issues
- Issues in multiple languages (indicates international user base = production reach)

**Experiment-only signals:**
- Issues exclusively from the repo owner or their forks
- No issues at all on a reasonably starred repo — either too new or nobody is actually using it
- All issues are feature requests, no bugs — real users generate friction; experiments don't
- Stale bot auto-closing issues with no maintainer engagement
- Issues closed with "won't fix" or "out of scope" on every user request

### 3.4 README quality — what makes claims extractable

A README that supports research claim extraction will typically have:

1. **Paper citation or arXiv link** — "Based on our paper [Title](arxiv.org/...)". This anchors claims to a peer-reviewed source. The repo implements; the paper makes claims.
2. **Benchmark table** — rows are datasets/tasks, columns are methods/models, values are metrics. This is the highest-value claim surface in a repo. Look for RESULTS, LEADERBOARD, or EVALUATION sections.
3. **Architecture diagram** — Mermaid diagrams or images showing component structure and data flow. Claims about system design live here.
4. **Explicit baseline comparison** — "Our method achieves X on Y dataset vs. Z for baseline W." Extractable quantitative claim with built-in context.
5. **Changelog or release notes** — Version history with dates. Claims that survived multiple versions have higher confidence than claims that appeared in v0.1 and were quietly removed.
6. **Usage section with real examples** — Evidence the repo does what it claims. Aspirational docs have usage examples that don't work.

RED FLAGS that lower claim trustworthiness:
- README is mostly installation instructions with no results or comparisons
- Claims reference "paper coming soon" without a preprint
- Benchmarks only on toy datasets (e.g., GSM8K without harder evals like SWE-bench or GAIA)
- "State of the art" claim with no citation to what baseline was beaten

## 4. Claim Extraction: Repositories vs. Academic Papers

### Core difference

**Papers:** claims are in Abstract, Introduction (contributions list), and Results/Evaluation. Structured, cited, peer-reviewed.

**Repositories:** claims are distributed — README, inline comments, docstrings, linked papers, benchmark scripts, issue discussions. Self-reported. May not match actual implementation. Often aspirational.

### Priority order for extraction

1. **Linked arXiv/paper** — if the README links to a paper, the paper is the authoritative claim source. The code is the implementation artifact. Extract from the paper; validate against the code. The two should agree on key claims; disagreements are themselves findings.

2. **README benchmark table** — most actionable. Check whether the benchmark is a known standard (SWE-bench, HumanEval, MMLU, AgentBench, GAIA, WebArena) or a custom eval. Custom evals require more skepticism — they may be designed to favor the approach.

3. `/results`, `/eval`, `/experiments` directory — raw numbers, scripts, or logs. More trustworthy than README claims because they include methodology. Look for `results.json`, `leaderboard.csv`, evaluation scripts with hardcoded comparison baselines.

4. **CHANGELOG / release notes** — what changed between versions reveals what actually works vs. what was aspirational. Claims that survived v0.1 → v0.5 have higher confidence.

5. **Issues and PRs** — user-reported limitations are often more accurate than author-reported capabilities. Search issues for: "doesn't work", "fails on", "worse than", "compared to". These surface counter-claims and scope limitations.

6. **Dependents page ("Used by")** — high dependent count is evidence the API is stable and claims were real enough to build on. A repo with 3k stars and 0 dependents has never been integrated into production code.

### Caution flags for extracted claims

- Self-reported benchmarks without reproducible scripts: aspirational
- Results only on training-adjacent datasets: potential data contamination
- Comparisons to cherry-picked or outdated baselines
- Claims that appear in README but have no corresponding code or script
- Version 0.x with no stable release: API and behavior may change significantly

## 5. Practical Search Patterns — Worked Examples

### Example 1: Finding agentic scaffolding frameworks

**Goal:** Discover production-used frameworks for building LLM agents with tool-use, memory, and planning.

```
# Step 1 — topic-first, high-precision
topic:agentic-framework language:python stars:>=1000 pushed:>2025-01-01 archived:false

# Step 2 — catch what topics miss (README-based)
"tool use" "planning" "memory" in:readme language:python stars:>=500 pushed:>2025-01-01

# Step 3 — find survey/comparison repos (high signal for claim extraction)
"agentic" "benchmark" "comparison" in:readme stars:>=200 pushed:>2024-01-01

# Step 4 — find implementations linked to papers
"agent" in:name language:python stars:>=100 topic:llm-agents pushed:>2025-01-01
```

**July 2026 results from query 1:**
`topic:llm-agent language:python stars:>=500 pushed:>2026-01-01` returns ~80 repos.
Top results by stars: autogen/59k (Microsoft, actively maintained), agentscope/27k (Alibaba, updated July 2026), letta/23k (stateful agents with memory).

**High-value research artifacts found via this pattern:**
- `FareedKhan-dev/all-agentic-architectures` — 35 production-grade architectures (Reflexion, LATS, MemGPT, Voyager) with benchmark table; 17-task leaderboard
- `tmgthb/Autonomous-Agents` — daily-updated paper list; use as index, not for claim extraction

Both surface via `topic:llm-agent stars:>=5000 pushed:>2026-01-01`. The `harness-sdk` name is a strong signal for kit-maker relevance — it uses the same "harness" terminology.

## 6. Known Failure Modes

### 6.1 Star inflation / fake stars

**Scale:** A peer-reviewed study by CMU, NCSU, and Socket (2024, expanded through 2026) found 6 million suspected fake stars across 18,617 repositories and 301,000 accounts. 78 repositories with detected fake-star activity appeared on GitHub Trending, confirming that purchased stars successfully manipulate recommendation algorithms.

**Detection heuristics (from the CMU study and June 2026 practitioner analysis):**

- **Fork-to-star ratio:** Healthy projects maintain approximately 90–235 forks per 1,000 stars. Below 50 forks per 1,000 stars on a repo exceeding 10,000 stars is a red flag. Forks require intent; stars require a single click.
- **Watcher-to-star ratio:** A legitimate repo shows roughly 1 watcher per 30–100 stars. A repo with 157k stars and 168 watchers (1:934 ratio) is almost certainly inflated.
- **Stargazer profile quality:** >30% of stargazers with zero followers, no repos, and no bio indicates purchased engagement. Legitimate repos: 1–2% ghost accounts. Flagged repos: up to 80% zero-follower accounts.

**Practical mitigation:** For any repo with >5k stars and no associated paper or well-known org, check the fork count and watcher count on the repo page. Use star-history.com to inspect growth trajectory — a single spike followed by 18 months of flat growth is a viral-moment project, not a sustained-use tool.

### 6.2 Default sort is "best match," not stars

GitHub's default search sort uses a black-box relevance algorithm. For discovery, always sort explicitly: append `sort:stars` to the URL or use the Sort dropdown. Without explicit sort, high-star repos that score lower on relevance may be buried below low-star repos that better match the keyword.

### 6.3 Archived repos included by default

GitHub includes archived repos in all searches unless excluded. Many deprecated AI projects remain indexed with high star counts. Always include `archived:false` in production research queries. Note: a bug reported December 2025 causes `is:archived` to behave inverted in some contexts — use `archived:false` (boolean form) rather than `is:archived`.

### 6.4 README content indexing lag

There is a latency between when README content is committed and when `in:readme` queries reflect it. New repos may not appear in README-based searches for hours to days after the initial commit.

### 6.5 Topic exact-match only — variant pair problem

`topic:llm-agent` does not return repos tagged `topic:llm-agents`. Common variant pairs that must be searched separately:

| Search this | Also search this |
|---|---|
| `topic:llm-agent` | `topic:llm-agents` |
| `topic:autonomous-agent` | `topic:autonomous-agents` |
| `topic:ai-agent` | `topic:ai-agents` |
| `topic:language-model` | `topic:large-language-model` |

### 6.6 Forks vs. originals

GitHub search returns forks alongside originals. Two failure modes:

1. **Counting the same work twice:** A popular repo forked into an org with 200 additional stars appears independently from the 10k-star original.
2. **Fork-as-primary confusion:** When an original is abandoned and a fork is actively maintained, the fork may outrank the original on `pushed:` filters but not on stars. The fork header on GitHub shows "forked from ORIGINAL/REPO" — check this when a repo's star count seems inconsistent with its recent activity.

### 6.7 "Awesome list" contamination

Many top results for AI topics are `awesome-X` curated lists (e.g., `awesome-llm-agents`, `awesome-agentic-reasoning`) rather than implementations. These are valuable for discovery and as indexes to implementations, but they contain no original benchmarks or research claims. Filter with `NOT awesome in:name` or recognize the pattern on sight. The `ARUNAGIRINATHAN-K/awesome-ai-agents-2026` (300+ entries) is typical — valuable as a map, not as a claim source.

### 6.8 Launch-day star burst distorts quality signal

A single Hacker News front-page appearance can produce 500–3,000 stars in 24–48 hours (documented in arXiv 2511.04453, Nov 2025), followed by plateau or slow decline. A repo with 2,000 stars and a single spike in star history + flat growth for 18 months is a one-viral-moment project, not a sustained-use tool. Star trajectory matters more than raw count for production-use assessment.
