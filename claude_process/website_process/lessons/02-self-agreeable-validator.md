# Lesson 02 — Self-Agreeable Validator

**The pattern**
Expected values in tests are produced by running the code and capturing whatever came out. The snapshot, fixture, or golden file is then frozen and treated as "the spec." From that point forward the test confirms only that the code still agrees with itself. Any bug present at capture time is now the standard. For instance, a team writing tests for their pricing-display component might run it once, copy the rendered `"$1,299.00"` into a snapshot, and never notice that the spec actually called for `"$1,299"` with no trailing zeros — the test passes forever. Or a transactional-email template whose expected HTML is the output of the first send, including a typo in the salutation that is now locked in by the fixture.

**Root cause**
Snapshots are convenient. Writing expected values from the specification requires reading the specification, translating it, and risking disagreement with the implementation. Capturing output bypasses that work. The validator inherits the producer's mistakes and then defends them.

**Rule adopted**
Expected values are derived from the spec, ticket, design, or external source of truth — written down before the code runs. Snapshot-style captures are acceptable only after a human has compared the captured output against the spec and signed off.

**How to recognize this in your project**
- Test fixtures were generated by `--update-snapshots` and never reviewed line-by-line
- PR diffs for "test updates" are large and approved without scrutiny
- Tests pass on broken behavior because the broken behavior was captured
- No traceable link from a fixture back to a spec, ticket, or design
- Changing the implementation is the standard way to "fix" a failing test
