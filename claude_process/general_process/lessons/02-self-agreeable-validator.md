# Lesson 02 — Self-Agreeable Validator

**The pattern**
Expected values in tests are produced by running the code and capturing whatever came out. The snapshot, fixture, or golden file is then frozen and treated as "the spec." From that point forward the test confirms only that the code still agrees with itself. Any bug present at capture time is now the standard. For instance, a team writing tests for a CLI's `--list` output might run it once, paste the stdout into a snapshot, and never notice that the spec actually called for tab-separated columns when the implementation produced space-padded ones — the test passes forever. Or a library whose API documentation is auto-generated from the implementation: a parameter's docstring says "milliseconds" because that's what the code uses, even though the design spec called for seconds, and the docs perpetuate the implementation bug indefinitely.

**Root cause**
Snapshots and auto-generated fixtures are convenient. Writing expected values from the specification requires reading the specification, translating it, and risking disagreement with the implementation. Capturing output bypasses that work. The validator inherits the producer's mistakes and then defends them.

**Rule adopted**
Expected values are derived from the spec, ticket, design, or external source of truth — written down before the code runs. Snapshot-style captures are acceptable only after a human has compared the captured output against the spec and signed off.

**How to recognize this in your project**
- Test fixtures were generated by `--update-snapshots` and never reviewed line-by-line
- PR diffs for "test updates" are large and approved without scrutiny
- Tests pass on broken behavior because the broken behavior was captured
- No traceable link from a fixture back to a spec, ticket, or design
- Changing the implementation is the standard way to "fix" a failing test
