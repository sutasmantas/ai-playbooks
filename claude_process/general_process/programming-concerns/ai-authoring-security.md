# Concern: AI-Authoring Security

**Rule**
Code generated by AI agents (or with significant agent assistance) is treated as untrusted until reviewed: scanned for known agent-introduced patterns (SSRF, weak crypto, hardcoded fallbacks, `eval` of external input, ReDoS-prone regex), and never merged on the strength of "the agent said it works."

**Use when**
Any project where an AI agent contributes code. In 2026, that's nearly all of them. Higher diligence on stake-tagged paths (auth, payments, deletion, pii — see `procedures/stake-tagging.md`).

**How to apply (skeleton — fleshed out in Slice 4)**
- Verifier checks agent-introduced code against the documented anti-pattern list (CWE-mapped flaws, OWASP LLM Top 10).
- Hardcoded fallback values (`"test"`, `"changeme"`, `"sk_live_AAAA..."`) are blocked at lint / PR / verifier time.
- SSRF-safe fetch wrapper for any outbound HTTP from user-controlled or LLM-suggested URLs.
- License hygiene NOT a concern (already settled in this kit's design — license compliance for AI-generated code is contested and unsettled; the kit does not enforce it).

**Skip if**
Truly never in a 2026 codebase.

**Project kinds applicable:** CLI, library, service, script, data-pipeline (ALL).
**Language-specific addendum:** PARTIAL — the patterns are language-agnostic, but the linters that catch them are not.

**Trace:** Veracode 2025 (45% of AI-generated code carries a CWE flaw); SSRF in all 5 major coding agents (Dec 2025 study); School of Reward Hacks (arXiv 2508.17511) on hardcoded-test contamination.
