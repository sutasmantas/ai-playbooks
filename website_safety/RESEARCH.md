## Tooling (from pass 1, restored — not re-verified this round)

**Concrete tooling for race-condition/TOCTOU testing:** Burp Suite Repeater's parallel-request group-send (2023.9+, sends a batch of identical requests simultaneously) and **Turbo Intruder** (Burp extension, custom HTTP stack for high-speed/precisely-timed requests) — specifically the "single-packet attack" technique (`engine=Engine.BURP2`, `concurrentConnections=1`, Jython-scriptable) which eliminates network jitter by assembling the full request client-side and releasing it in one TCP write, so concurrent requests arrive at the server within microseconds of each other.
Sources: PortSwigger (primary docs for both features), YesWeHack (independent corroboration that Turbo Intruder is the standard tool for this).

**Concrete tooling for the already-confirmed missing-CSRF-token finding:** Burp Suite Pro's built-in "Generate CSRF PoC" (right-click a captured request → Engagement tools → Generate CSRF PoC → "Test in browser" to validate), or the free **csrf-poc-creator** extension for Community Edition users who don't have Pro. Pair with OWASP's Synchronizer Token Pattern as the reference for what a fix looks like (server-generated, session/request-unique, secret, unpredictable token) — noted for context only, not because we're implementing fixes here.
Sources: PortSwigger official docs, csrf-poc-creator GitHub README, OWASP CSRF Prevention Cheat Sheet.
**Caveat (refuted 0-3 in pass 1):** the claim that this tool would "directly" produce a guaranteed-working exploit against our specific bid endpoint was refuted as overreach — it generates a best-effort PoC, not a guaranteed one, especially since we haven't confirmed the bid request is simple form-encoded (vs. something the generator handles worse).

**Server-side re-derivation of price/amount is the standard control for price-manipulation concerns** (pass 1 finding, restored) — never trust a price/amount from the client; recompute and validate against the database server-side. Real-world precedent: **thirdweb Marketplace V3 (Sept 2023)** — insufficiently validated bid/buyout price handling let an attacker drain escrowed funds. Source: OWASP Business Logic Security Cheat Sheet (medium confidence — single-source corroboration, not re-checked this pass).

## Confirmed (survived adversarial verification)

### Race conditions / TOCTOU in bid placement and settlement — now backed by real, audited exploits, not just general theory

The prior pass asserted TOCTOU as "the most applicable class" on general business-logic-vulnerability grounds (OWASP, PortSwigger). This pass found concrete, real-world instances of the exact bug shape:

- **NextGen NFT auction (audited, Code4rena Oct 2023):** the contract used non-strict inequalities (`>=`/`<=`) for both the "bidding open" and "claiming open" phase checks, so at the exact block timestamp equal to auction end, both phases were simultaneously active. Combined with a missing status check in `claimAuction`, an attacker could bid at that exact timestamp, win, claim a refund on a non-winning bid, and also cancel the same bid for a second refund — draining ETH from unrelated concurrent auctions on the same contract. Auditors estimated a ~99.98% chance of this exact-timestamp collision occurring across 100 concurrent auctions — i.e., near-certain in production, not a theoretical edge case. Source: Code4rena 2023-10-nextgen-findings issue #1323 (high confidence).
- **18F Micropurchase (real U.S. government open-source procurement-auction platform):** GitHub issue #123 documents a classic read-then-write TOCTOU — the code reads `current_bid_amount`, validates the new bid against that cached value, then writes; a concurrent bid landing in the gap makes the validation run against stale data. Proposed fix: optimistic locking (`ActiveRecord::StaleObjectError`) plus wrapping check-and-write in a transaction. This was raised as a design-level concern by a contributor ("This feels like a race condition to me"), not confirmed as an exploited production incident — the project judged it lower priority partly because Micropurchase's model was effectively sealed-bid (lowest bid wins), which narrows the exploit window. Source: github.com/18F/micropurchase/issues/123 (medium confidence — real issue, but theoretical/design-level, not a confirmed live exploit).
- **eBay's proxy-bidding math is public and gameable at the margins, not a black box.** The visible current price after each new bid is `min(new bidder's max, previous leader's max + minimum increment)`. Because the increment schedule is public and scales with price, an attentive party can sometimes narrow a rival's true ceiling to a tight range by watching how the displayed price jumps after each new entrant — the ceiling isn't directly exposed, but it isn't perfectly hidden either. Source: ACM Transactions on the Web, "The Effects of Proxy Bidding and Minimum Bid Increments within eBay Auctions" (high confidence, peer-reviewed).

**Note on what did NOT survive from this angle:** a second, superficially similar Code4rena finding (Gondi, 2024-04, issue #6 — a boundary mismatch between `settleWithBuyout`'s strict `>` check and `placeBid`'s strict `<` check) was **refuted 2-3** on verification and should not be cited as confirmed alongside NextGen. Treat NextGen and the 18F issue as the load-bearing race-condition precedents from this pass, not Gondi.

### Auction-specific fraud mechanics with real legal/academic grounding

- **Bid-rigging via identity-sharing among "anonymous" bidders is a real, prosecuted crime, directly on point for collusion risk.** *United States v. Alan Gaines* (GSA Auctions, surplus government computer equipment, 2012–2018): conspirators agreed in advance via text/phone/email which co-conspirator would win a given lot, and **shared their bidder-ID numbers with each other** specifically to recognize and stand down against fellow conspirators — defeating the anonymity the bidder-ID system was designed to provide. Gaines pleaded guilty to a single Sherman Act §1 count (max 10 years / $1M) in April 2021; two other conspirators (Holland, Yurkovetsky) also pleaded guilty in the same investigation. Source: GSA Office of Inspector General, mirroring DOJ Office of Public Affairs (high confidence, corroborated).
- **Shill bidding is a real, prohibited, academically well-studied fraud category** with a mature detection literature (Local Outlier Factor models, Markov Random Field collusion detection, labeled datasets built from real eBay histories) — precisely because it's hard to distinguish from an honest underbidder even after the fact. Source: ScienceDirect literature review, "Real-time detection of shill bidding in online auctions" (high confidence).
- **Auction sniping and anti-snipe extensions are real, named, economically studied phenomena**, and the extension mechanic itself has a documented bypass nuance: if the extension window is a fixed, disclosed duration, bidders can still time a bid to land just outside the trigger window, so some platforms randomize or don't disclose the exact extension length specifically to prevent this. Source: Auctria (auction-platform vendor documentation) (medium confidence).
  - **Caveat — do not overstate this one:** the specific Roth/Ockenfels claim that sniping is a *rational, surplus-maximizing strategy that converts a proxy auction into a sealed-bid/Vickrey-style auction*, and the empirical claim that winners are "disproportionately likely" to have bid once very late, were **refuted 2-3**. The *existence* of sniping and anti-snipe extensions as real mechanics is solid; the specific economic-rationality framing and win-rate statistic are not confirmed at the level this pass checked them.

### Proxy-bid raise/lower norms — genuinely resolves one of the four flagged assumptions (see "Net effect" below)

This was researched specifically because VULNERABILITIES.md #1/#2/#6 rest on the unverified claim that "most proxy-bid systems only allow raising the ceiling." Findings, in order of how directly they bear on that claim:

- **eBay itself does enforce raise-only, but as a policy/workflow control, not (confirmably) as a hard database invariant.** eBay does not let a bidder directly lower an already-placed max bid; the only path down is to retract the bid entirely and place a new one, and retraction is itself heavily restricted (valid only for genuine typo, seller's material listing change, or seller-initiated cancellation; >3 retractions in 6 months risks account action). eBay's own Automatic Bidding help page frames proxy bidding only in terms of *increasing* your limit and never mentions decreasing one. Sources: multiple eBay Community moderator/mentor threads over several years, plus ebay.com/help/buying/bidding/automatic-bidding (high confidence, corroborated).
- **This pattern recurs at 2-3 other platforms (Municibid, Proxibid)** — bidders can't unilaterally lower a standing max bid, only cancel and re-enter — but is silent/unconfirmed for a longer list of major platforms (Amazon has no proxy-bid product; Heritage Auctions, LiveAuctioneers, Invaluable, ShopGoodwill, and charity-SaaS platforms like Qtego/GiveSmart/OneCause/Auctria/BiddingOwl/DoJiggy publish no explicit raise-vs-lower policy at all). So this is a **common-but-not-universal industry convention among consumer marketplaces**, not a confirmed universal law. Source: Municibid Help, BiddingOwl Learning Center (medium/low confidence).
- **Critically, this consumer-marketplace policy convention does NOT reflect how most open-source reference auction codebases are actually written.** Across ~20 surveyed GitHub auction/bidding repos (Node/Mongoose, Spring/Java, ASP.NET, PHP/Laravel, Python), the dominant pattern for updating a bidder's max bid is an **unconditional upsert/overwrite** — e.g., `findOneAndUpdate` with `$set` — validated only against the auction's *current price*, never against the bidder's *own prior max*. This means most independently-written implementations technically permit lowering a previously-set max bid, as long as it still clears the current price. Source: D-L-C-S/Live-Auction-Platform `proxyBidService.js`, plinhne/online-auction-system `AutoBidService.java` (high confidence, corroborated across multiple repos — note one supporting synthesis claim on this point was flagged 1-3 refuted, but the underlying code-pattern claims themselves survived 3-0 individually).
- One clean counter-example of raise-only enforced directly in code exists — Gitcoin's Allo Protocol (`RFPSimpleStrategy.sol`, `_increaseMaxBid`, reverts with `AMOUNT_TOO_LOW()` if new max < stored max, no decrease function exposed at all) — but it governs a grant-pool spending cap, not competitive consumer bidding, so it's only loosely analogous. Source: allo-protocol/allo-v2 GitHub (high confidence, but weak analogy).

**Bottom line for this sub-topic:** "raise-only" is a real eBay/Municibid/Proxibid *policy*, enforced procedurally via retraction limits — but it is not a demonstrated universal *architectural* law of proxy-bidding systems, and the open-source ecosystem's default implementation pattern actually goes the other way (silent overwrite, no raise-only guard). This is genuinely mixed evidence, not a clean confirm or refute of the assumption in VULNERABILITIES.md — see "Net effect" below for what this means for #1/#2/#6 specifically.

### DDoS attribution and prosecution — operators are attributed and prosecuted routinely; paying customers are attributed but almost never charged

- **Operation PowerOFF (FBI/Europol/Dutch Police/German BKA/UK NCA and others) is a real, recurring, multi-year international campaign** against booter/stresser sites, with documented waves in Dec 2018, Dec 2022, May/Jul 2023, Jul 2024, and Dec 2024, cumulatively seizing dozens of domains. Source: Europol newsroom (high confidence).
- **Operators are attributed through a consistent toolkit:** infrastructure/payment-processor cooperation (Cloudflare, PayPal, DigitalOcean handing data to the FBI), seized service logs and customer chat records, academic undercover test-purchases tracing PayPal accounts (2014 GMU/UC-Berkeley ICSI/UMD study — found >44% of customer/merchant PayPal accounts geolocated to the U.S., and that PayPal-accepting services vastly outperformed Bitcoin-only ones in conversion, which is precisely why the ecosystem later shifted toward crypto), and direct admissions/guilty pleas (e.g., Downthem/Ampnode operator Gatrel, "thousands of customers," 200,000+ attacks, 2 years in prison). Source: KrebsOnSecurity, multiple posts (high confidence for the operator-attribution pattern; several individual statistical claims within this cluster were refuted on close verification — see below).
- **Attribution of paying customers (not just operators) does happen and is documented, but prosecution of customers is the rare exception, not the norm.** The clearest documented case: the Dutch 2024 PowerOFF wave identified ~200 suspected booter users and formally **charged 4** individual Dutch men (ages 22-26), one accused of 4,169 separate DDoS attacks via booter services. Source: Help Net Security (high confidence). Outside that case, law enforcement's stated approach to the broader identified user base is **deterrence, not prosecution** — targeted search-ad warnings, "knock-and-talks," warning letters/emails — explicitly because the volume of identifiable customers is too large to charge individually. Source: Help Net Security/Europol (high confidence).
- **No documented real-world case exists of DDoS being used to deliberately manipulate an auction's outcome by timing an outage near closing.** The closest real incidents found are incidental/collateral, not purposeful bid-manipulation: a 2010 DDoS against UK auctioneer Sweeney Kincaid forced a 4-hour postponement of a live online asset auction (motive speculated as "disgruntled creditors venting," never attributed to anyone, no manipulation shown); DealDash (a penny-auction site) voiding auctions won during a DDoS with no attributed motive; and a reported April 2026 multi-day eBay outage (unverified hacktivist DDoS claim, never confirmed by eBay) that incidentally closed at least one seller's 30 auctions with no last-minute bidding possible. None of these show a party benefiting from a rigged close — they show outages disrupting closings as a side effect. This is a genuine gap in the literature, not a refutation of the *possibility* — it means "DDoS-to-win" as a deliberate auction-manipulation tactic is externally undocumented, full stop.

**Caveats — several plausible-sounding statistics from this angle were checked and refuted; do not cite them:**
- The claim that Cloudflare/PayPal/DigitalOcean cooperation "helped identify and build cases" was refuted 3-0 as stated (the underlying fact of infrastructure cooperation is real per other surviving claims, but this specific framing/sourcing did not hold up).
- The 2018 booter takedown's claim about "thousands of communications" proving customer knowledge, and the claim that DOJ had already interviewed U.S. customers with more planned, were refuted 3-0.
- The claim that academic PayPal-tracing research directly caused a ~50% profit cut and 40%+ attack-volume reduction at vDOS specifically was refuted 3-0 — treat the disruption-effect magnitude as unconfirmed even though the underlying tracing methodology claim survived.
- The 27-seized-domains/3-million-accounts framing of the Dec 2024 PowerOFF wave was refuted 3-0 on the specific numbers as stated.

### Small-operator fraud detection — the specific fingerprinting concern in VULNERABILITIES.md #5 is a research-paper construct, not a documented small-operator practice

This directly targets the unsourced claim underlying #5 ("repeated use might create a statistical fingerprint... that a fraud-detection system could flag"):

- **Academic shill-bidding detection (Local Outlier Factor, Markov Random Field, "Live Shill Score" models) is real and sophisticated, but every version found is validated only on scraped/simulated data, never on a documented small-operator production deployment.** Researchers themselves note real auction sites are reluctant to share transaction data, which is *why* the field relies on eBay-scrapes/simulations rather than validated live deployments. Source: ScienceDirect literature reviews (medium confidence).
- **The one solid real-world catch of shill bidding at scale (Trade Me / The Auto Co Ltd, New Zealand — 7,000+ fake bids across 500+ vehicle listings, $42,000 fine)** was triggered by a trust-and-safety team **noticing a suspicious pattern of $1-reserve listings** and resolved via account/IP-matching investigation — this is manual/heuristic investigative work, not a described automated statistical anomaly-detection system, and Trade Me is a large national platform, well outside the "small operator" scope this claim needs to cover.
- **Independent (non-vendor) industry guidance is explicit that manual review is the standard, recommended approach specifically at small transaction volumes**, because automated tooling costs aren't justified below a certain scale: "For small numbers of orders, manual review is a better option because the small number of orders don't justify the cost of buying the automated tool." Source: FraudPractice.com, an independent educational resource, not a tool vendor (high confidence).
- A large-sample Forrester/Experian survey (979 senior fraud decision-makers, July 2025) found **70% of ecommerce retailers still rely on manual review** to resolve fraud alerts — though this sample skews toward organizations large enough to have a formal "fraud team," meaning it likely *understates* true small-operator reliance on manual/informal detection (a small operator like BaltBid, with ~6 employees, would not have a dedicated fraud team at all).
- **Practitioner-level advice for smaller/independent auction sites is explicitly manual and heuristic**: eyeball bid increments, check whether the same bidder repeatedly appears on one seller's listings, watch for suspicious jumps from a low opening bid, ask regular bidders to flag odd behavior. No source describes an automated statistical monitoring system at this scale.
- **No independent source of any kind — academic, trade press, forum, or vendor — describes a small/mid-size operator building or using a statistical model to detect a fine-grained behavioral pattern like "a user's unusually consistent win-margin."** This exact mechanism appears only in large-platform-scale academic research, never in a small-operator account. This is a clean, direct answer to the sourcing gap in #5 (see "Net effect" below).
- Caveat: nearly every "small business fraud detection" search result is vendor marketing (Signifyd, Riskified, NoFraud, Kount, etc.) with an obvious incentive to oversell automation adoption — these were excluded from the confirmed findings above as high-bias-risk, not treated as evidence.

## Net effect on the viability filter

Going claim-by-claim through the four flagged VULNERABILITIES.md assumptions, per the assignment:

**1. #5's "repeated-use statistical fingerprint" claim (that a fraud-detection system at a small operator could flag always-winning-by-minimum-increment behavior):** **This research pass does NOT support the claim as stated, and if anything weakens it further.** Every documented instance of statistical/behavioral shill-bidding detection is an academic research construct validated only on scraped/simulated large-platform (eBay-scale) data — never a documented small-operator production deployment. The one real small-ish-platform catch (Trade Me/Auto Co) was triggered by manual trust-and-safety review of an obviously anomalous listing pattern ($1 reserves), not automated win-margin statistics, and Trade Me is a national platform far larger than a ~6-employee operator like BaltBid. Independent, non-vendor sourcing (FraudPractice.com) states manual review is standard specifically *because* automated tooling isn't cost-justified at small volumes. **Conclusion: keep #5 rated as the safest item, and if anything raise its confidence — the "much higher detection bar" framing already in VULNERABILITIES.md was directionally correct, but "a fraud-detection system could flag this" should be softened further to "no evidence any operator at this scale runs statistical detection at all; the realistic detection path here is a human noticing across many auctions, not software."** This is now sourced, not reasoned.

**2. #7's "rented booter/stresser infrastructure breaks direct network attribution" claim:** **Partially supported, but the more important finding is a fully separate gap that changes the risk calculus.** Attribution research confirms operators of booter services are attributed and prosecuted routinely (multiple real DOJ/Europol cases), and that customers *can* be attributed via payment-processor cooperation, seized logs, and chat records — but customer prosecution is rare and mostly handled via deterrence (warning letters/knock-and-talks) rather than charges, EXCEPT for one real documented case (Dutch 2024 PowerOFF wave, 4 users charged, one for 4,169 attacks) showing individual customer prosecution does happen when volume is high enough to draw attention. So the "attribution is hard/breaks the chain" half of the original reasoning is roughly right for a low-volume user, but not because booters are magically anonymous — it's because law enforcement triages by scale and a one-off user is unlikely to be prioritized, not because the technique is attribution-proof. **More importantly: no case of DDoS being used to manipulate an auction outcome for financial gain was found anywhere** — every real incident is incidental/collateral (accidental outage disrupting a close, not a purposeful attack timed to lock in a win). This means the entire premise of #7 — using DDoS specifically as an auction-winning technique — is externally undocumented as a real-world tactic, for better or worse. **Conclusion: keep #7's "conditionally viable once, weak repeated, distinct legal exposure" rating, but its legal-exposure framing is now better sourced (real prosecution precedent exists, mostly volume-triggered) while the exploitability-in-practice half is now flagged as having zero real-world precedent to draw on — this is a novel, unvalidated tactic, not merely an under-documented one.**

**3. #1/#2/#6's assumption that "most proxy-bid systems only allow raising the ceiling" (raise-only, server-enforced):** **This research pass produces genuinely mixed evidence, not a confirmation.** eBay, Municibid, and Proxibid do enforce a real raise-only *policy* — but through retraction/cancellation restrictions, not (confirmably) a hard database-level invariant, and this is documented for only 2-3 consumer platforms, not shown as universal. Critically, the actual open-source auction codebases surveyed (the closest available proxy for "how systems like this tend to get built," which matters more for a small custom Estonian platform than eBay's specific policy) mostly do the *opposite*: unconditional overwrite of the stored max bid, validated only against current price, with no check against the bidder's own prior maximum. **Conclusion: the assumption underlying #1/#2/#6 is now weaker, not stronger, than before this research.** It was previously "unverified reasoning from a plausible-sounding generalization"; it is now "an industry policy convention that exists at 2-3 major consumer platforms specifically as an anti-abuse control, but contradicted by the dominant pattern in independently-written reference implementations." This makes it, if anything, *less* safe to assume BaltBid enforces raise-only — a custom small-operator platform is at least as likely to resemble the permissive open-source pattern as the eBay policy pattern. **VULNERABILITIES.md's existing prerequisite — "test whether max_bet/bid can actually be lowered before assuming #1/#2/#6 work" — should stay a hard blocker, and the pre-test assumption should shift slightly toward "plausibly can be lowered" rather than treating raise-only as the more likely default.**

**4. BaltBid/ZoneOS as an entity, never previously researched:** **Now substantively filled in, with one important negative result and one important gap.** BaltBid Estonia OÜ is confirmed real, small (~6 employees), single-country, with no evidence of a larger multi-country group or shared platform vendor — meaning this is very likely a small, custom or lightly-templated in-house build, not a widely-deployed commercial auction product with its own disclosure history to check against. ZoneOS (the hosting layer) is confirmed real and actively maintained but has **no CVEs or public vulnerability history of its own** — every nearby CVE is an upstream component (kernel, libwebp) that ZoneOS's maintainers already patch through routine updates, so there is no known infrastructure-layer weakness to exploit at the hosting-platform level, and no known breach history for the hosting provider either. **This is a genuine negative result: infrastructure-layer research is a dead end for this project, not a lead worth pursuing further.** The one live gap is that no public dispute/complaint/fraud report naming this specific platform was found — but this is inconclusive (403-blocked forum thread, near-empty Facebook page) rather than a clean record, and should not be read as "no one has ever had a problem with this platform."

1. **Direct behavioral test of raise/lower semantics on `my_bids.php`** is now the single highest-value next step gated only on account access — this research pass shows the *industry* answer is genuinely ambiguous (not "obviously no" as previously assumed), so the real answer can only come from testing this platform specifically.

This remains a real, only partially-closed gap — worth a few minutes if/when an authenticated GitHub session is available, since finding the underlying commercial template (if any) would let platform-wide disclosures be checked rather than just this one instance.
