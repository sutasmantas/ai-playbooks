# Attack Catalog — avariilised-autod.ee

## Scope of this whole document — read this before adding or re-prioritizing anything

This catalog is deliberately filtered by **two independent requirements, both must hold**, not just one:

1. **Monetary, from manipulating the auction itself.** The attacker (a bidder on the platform) must end up gaining money — winning an auction they shouldn't, winning at a lower price than genuine competition would produce, or extracting information that gives them an unfair pricing edge. Pure griefing/harm-to-others with no gain to the attacker is excluded. Data exposure that doesn't touch a live auction's outcome (e.g. the confirmed `auction_bids.php` leak of *past, settled* auctions — see FINDINGS.md) is tracked but is explicitly **not** what this catalog is optimizing for.
2. **Safe for the attacker to actually use.** Every account on this platform is KYC'd (real name, national ID number, phone, address, a bank transfer at registration) — there is no anonymous-account option here. So a bug only counts as a *real, usable* money-vulnerability if a rational attacker could exploit it without the result screaming "fraud" to whoever reconciles sales. An exploit that produces a visibly implausible outcome is self-reporting regardless of how the exploit itself works — see the "Attacker viability filter" section below for the full reasoning and the per-item verdicts.

**If you're a different agent picking this up: the `[x]`/`[ ]`/`[-]` status marks below are about whether a bug *technically exists*. The `VIABLE`/`CONDITIONALLY VIABLE`/`NOT VIABLE` tags are a separate axis about whether a real attacker would actually risk using it. Prioritize testing effort by the second axis, not just the first — see "Net effect on priority" at the end of the viability section.**

Status legend: `[ ]` not tested · `[x]` confirmed present · `[-]` tested, not present

## Attacker viability filter (2026-07-01) — would this actually get used without blowing up on the attacker?

**User's framing, generalized as the governing test for this section:** a bug is only a *usable* money-vulnerability if an attacker could exploit it and have the result look like a normal outcome. Registration on this platform requires real name, national ID number, phone, a physical document number, address, and a bank transfer — every account is KYC'd. So "the attacker gets caught" here doesn't mean some anonymous internet stranger vanishes; it means the platform/seller flags an implausible result, investigates, reverses the win, and already has the attacker's real identity on file. That's a fundamentally different risk profile from typical anonymous e-commerce fraud, and it changes which bugs are actually worth worrying about.

**The dividing line that matters:** does the exploit produce a *visibly implausible outcome* (a car selling for a few euros, a win with no real bid history, a sale nobody can explain), or does it just quietly change *which otherwise-normal-looking price* the auction settles at? The first kind is self-reporting — no one needs to find the bug, the absurd price finds them. The second kind is invisible from the seller's side because there is nothing to compare it against: nobody can prove what the "honest" price would have been.

**Self-audit, 2026-07-01 (after the #7/DDoS correction) — same bias found in three other places.** The DDoS mistake wasn't really about DDoS specifically — it was two generic reasoning errors: (1) assuming the attacker uses an exploit in its *most extreme, most conspicuous form* rather than a tunable/subtle one, and (2) asserting "no way to ever detect this" when the honest claim is "not normally reviewed, but not literally invisible if anyone ever looked closely." Went back through every item checking for the same two errors. Found real corrections in #3, #4, #5, #3b, and an unverified assumption underlying #1/#2/#6 that was never actually confirmed. #8, #9, #10 held up.

**External research pass, 2026-07-01 (RESEARCH.md second pass, 7 angles, 3-vote adversarially verified) — grounds four of the reasoning-only claims above in real sources.** Net effect, in order of how much it moved things: (1) #5's fraud-detection caveat should be **softened further, not just kept** — no source of any kind describes a small operator running statistical win-margin detection; the real risk is a human noticing across many auctions, not software, and this is now sourced rather than reasoned; (2) #1/#2/#6's "raise-only" assumption comes out **weaker than before**, not stronger — real open-source auction codebases mostly allow overwriting a stored max bid unconditionally (no check against the bidder's own prior max), so the pre-test assumption should shift toward "plausibly can be lowered," while eBay/Municibid-style raise-only is a real but platform-specific *policy*, not a universal architectural law; (3) #7's booter-attribution reasoning is roughly right but for a different reason than originally stated — law enforcement attributes and prosecutes booter *operators* routinely, and *can* attribute paying customers, but mostly triages by volume rather than charging individuals, which is why a one-off use is lower-risk — and, separately, **no real-world case of DDoS being used to manipulate an auction outcome was found anywhere**, meaning the tactic itself is genuinely novel/unvalidated, not just under-documented; (4) #3b gets real external backing for the first time — an **audited, real smart-contract exploit (NextGen NFT auction, Code4rena Oct 2023)** hit the identical bug shape (inclusive/exclusive timestamp-boundary mismatch between the bid-open and claim-open checks), auditors estimating ~99.98% collision odds in production — this is a materially stronger precedent than the single thirdweb citation from pass 1. Full detail in `RESEARCH.md`.

## 1. Eliminate a competing bid via forged request (CSRF) — `CONDITIONALLY VIABLE, RESTS ON AN UNVERIFIED ASSUMPTION`

Mechanism: `POST /auctions/my_bids.php` (`auction_id`, `bid`, `max_bet`) has no CSRF token.
- [x] Confirmed: no token on bid or max_bet submission.
- [ ] Confirm GET-acceptance (`$_REQUEST`) — if yes, a plain link sent to a competing bidder can silently zero out their max auto-bid or otherwise neutralize their standing bid, removing them as competition so the attacker's own (lower) bid becomes the winner.
- [ ] **Not yet confirmed — test this before assuming the attack works at all:** does the backend actually allow a `bid`/`max_bet` value to go *down*? Most proxy-bidding systems only allow raising your ceiling (server-enforced monotonic increase) specifically to prevent this. If that's true here, CSRF can't "neutralize" a rival's bid this way regardless of the missing token.

## 2. Hijack a competitor's session to withdraw their bid — `CONDITIONALLY VIABLE, RESTS ON THE SAME UNVERIFIED ASSUMPTION AS #1`

Mechanism: session cookie has no `HttpOnly` (confirmed) — any XSS anywhere on the site (#4) lets an attacker read `document.cookie` and hijack a rival bidder's session, then use it to cancel/lower their bid or max_bet, clearing the way for the attacker's separate account to win cheaper. Same open question as #1: unconfirmed whether the backend permits lowering a bid/`max_bet` at all.

## 3. Bypass server-side bid validation to register an artificially low winning bid — `SPLIT VERDICT: EXTREME FORM NOT VIABLE, SMALL FORM CONDITIONALLY VIABLE` (revised 2026-07-01 — see attacker-viability filter above)

- [ ] Does the server accept a bid below current price + minimum increment? **Test the boundary, not just yes/no** — a bid landing far below true value ("selling" for a few euros) is a screaming, KYC-traceable anomaly (not viable); a bid landing only modestly under the correct increment (a small, explainable-away shortfall) doesn't carry the same guaranteed scrutiny and is the more realistic thing to actually look for.
- [ ] Does the server accept a bid after the posted end time, or fail to enforce the anti-snipe extension? A snipe that still wins at a normal-looking price (just via a technicality on timing) is closer to the "small/plausible" bucket above than the extreme one.
- [ ] Negative/zero/non-numeric/overflow bid amount handling — these would produce an obviously-broken result (negative price, etc.), so stay in the not-viable bucket regardless of what's found.

## 3b. Race condition / TOCTOU on bid placement (research-confirmed as the most applicable business-logic class here)

Any read-current-price → decide-if-valid → write-new-bid sequence is a potential race condition if not atomic. Concurrent identical or near-identical requests can each read the same stale "current price," each independently pass validation, and each get committed — meaning the true winner isn't necessarily whoever the UI shows as having "won."
- `VIABLE, THOUGH NOT LITERALLY UNAUDITABLE` — [ ] Fire two (or more) simultaneous bid requests at the same price point / same auction and see whether more than one gets accepted, or whether the accepted one is nondeterministic in a way that could be gamed (e.g. always favors whichever request the server happens to process last, which an attacker could exploit by firing last on purpose). This form doesn't produce a below-market price — the auction still settles at a normal, explicable amount, just with a different winner among comparably-priced bids. Low baseline detection risk since no one reviews request-ordering logs by default, but scripted/automated requests can look different from organic browser traffic if a rival ever disputes a tie-break specifically and someone pulls raw logs.
- `NOT VIABLE` — [ ] Combine with the confirmed missing-CSRF-token gap (#1): a forged cross-site request timed precisely against a race window could theoretically let an attacker win a stale-price check on a victim's session or the attacker's own, simultaneously bypassing two controls at once. This variant, if it worked, would let the attacker win *below* the true current price — the same screaming, KYC-traceable anomaly as #3, just reached via a different mechanism. Not yet tested — needs precise request timing (see RESEARCH.md tool notes on Turbo Intruder/single-packet attack).
Real-world precedent: thirdweb Marketplace V3 (Sept 2023) — a bid/buyout marketplace with insufficiently validated price handling let an attacker drain escrowed funds. Closest documented case to this exact scenario found in research.

## 6. Directly remove a competitor's bid (IDOR) — `CONDITIONALLY VIABLE, SAME UNVERIFIED ASSUMPTION AS #1/#2`

## 7. Win by taking the bidding mechanism offline while holding the high bid — `CONDITIONALLY VIABLE ONCE, WEAK REPEATED, DISTINCT LEGAL EXPOSURE` (revised 2026-07-01 — see the viability filter table above for the full reasoning on why the original "not viable" call was wrong)

Mechanism: no CDN/WAF observed in front of origin (confirmed); no rate limiting observed on the bid endpoint or the public polling endpoint at 15 rapid requests (confirmed — all returned identical status codes, no throttling).
- Attacker places a **plausible, not-obviously-fake** leading bid, then floods the bid endpoint (or the whole site) — via rented/booter infrastructure, not their own connection, to avoid direct network attribution — so no competing bid can be submitted before the auction's scheduled close. Doesn't require winning at an absurd price (that would fall back into the same trap as #3/#8); just needs bidding suppressed while already holding a bid that looks like it could have won organically anyway. One-off use has a cover story ("the site had an outage"); repeated use to make this an actual income source reintroduces a detectable pattern (same account, same story, every time).

## 8. Exploit locale-dependent number parsing to register a misparsed low bid — `NOT VIABLE` (same screaming-price-anomaly problem as #3)
