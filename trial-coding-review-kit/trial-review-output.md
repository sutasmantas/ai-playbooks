### **P1 correctness — args is accessed without null guard**

Severity: P1 — TypeError at runtime if workflow is invoked without args; surfaces only in misconfiguration but the failure is cryptic and unrecoverable mid-workflow
Category: correctness
Blocking: YES

Root cause: Lines 75–79 access `args.angles`, `args.archivePath`, `args.domain`, and `args.openDecisions` before any null check. In the workflow system, `args` is a global that is `undefined` when the workflow is launched without an args parameter. The first access `args.angles` (line 75) would throw `TypeError: Cannot read properties of undefined`. `args.openDecisions && args.openDecisions.length` (line 79) would also throw — `args.openDecisions` is evaluated first, and if `args` is undefined this evaluates `undefined.openDecisions` which throws before the `&&` short-circuit applies.

Finding: `const angles = args.angles` (line 75) — no null guard. If this workflow is invoked as `Workflow({script: "..."})` without the `args` field, the entire workflow fails immediately at line 75 with no helpful error message.

Fix:
```javascript
if (!args || !args.angles || !Array.isArray(args.angles)) {
  throw new Error('research-wave: args.angles (array of {title, queries}) is required')
}
const angles = args.angles
```

Counter-check: Is `args` actually guaranteed to be defined by the workflow system? Check: the workflow tool description says `args` is `undefined` if not provided — confirms the bug.

### **P1 reliability — newAngles collected but never written to RESEARCH_ANGLES.md**

Severity: P1 — research angles generated by waves are permanently lost unless the human caller reads the return value and manually acts on the instructions string
Category: reliability
Blocking: YES

Root cause: Lines 164–172 collect `allNewAngles` from each angle result. Lines 334 include them in the return summary. Line 334 also includes `instructions: "Then add newAngles to RESEARCH_ANGLES.md"` — but this instruction is a dead string: no agent, no tool call, and no procedure writes to RESEARCH_ANGLES.md. The caller receives a return object with new angles embedded in a nested structure and an instruction string, but the caller must independently implement this step. The kit-maker's post-cluster-review procedure does not reference this return field.

Finding: Every research wave produces new angles (surfaced in `return.summary.newAngles`) that systematically fail to reach RESEARCH_ANGLES.md unless a human manually reads the return value and adds them. This defeats the purpose of the `newAngles` pipeline.

Fix: Add an Archive phase step that writes new angles to RESEARCH_ANGLES.md:
```javascript
if (allNewAngles.length > 0) {
  await agent(
    `Append these new research angles to RESEARCH_ANGLES.md at path ${archivePath}/../RESEARCH_ANGLES.md.
    Format each as: "## [title] (priority: [priority])\n[wouldFind]\n\nGrounding target: [groundingTarget]"
    Angles to append: ${JSON.stringify(allNewAngles, null, 2)}
    Use the Edit tool to append to the file.`,
    { label: 'write-new-angles', phase: 'Archive' }
  )
}
```

Counter-check: Could the caller reliably handle this themselves? Check: the instructions field is a plain string; callers have no structured way to know they must act on it. Confirmed unreliable.

### **P2 correctness — pipeline() used for synchronous transformation in archive phase**

Severity: P2 — architectural mismatch; does not crash but misleads readers and may have subtle behavior with sync stages in pipeline()
Category: maintainability
Blocking: NO

Root cause: Lines 233–309 use `pipeline(angles.map(...), ({ angle, result }) => { ... })` where the single stage is a synchronous data transformation — no `await`, no agent calls, no I/O. `pipeline()` is designed for async multi-stage agent pipelines. Using it for synchronous data transformation is an architectural category error.

Finding: `const archiveFiles = await pipeline(angles.map(...)` (line 233) — single sync stage, no async operations. This should be a `.map()` call: `const archiveFiles = angles.map((angle, i) => { ... })`.

Fix: Replace `await pipeline(angles.map(...), stage)` with `angles.map((angle, i) => { const result = angleResults[i]; ... })`.

Counter-check: Does pipeline() handle sync stages correctly? The workflow system likely awaits the stage even if it returns a non-Promise. Behavior is likely correct — this is a code clarity issue, not a functional bug. Severity confirmed as P2, not P1.

### **P2 reliability — archive phase never writes files; phase label is misleading**

Severity: P2 — workflow meta and phase label ("Archive — Write structured markdown") describe writing behavior that doesn't happen inside the script; the write is silently delegated to the caller
Category: reliability
Blocking: NO

Root cause: `phase('Archive')` (line 230) and `meta.phases[2].detail: 'Write structured markdown to research_archive/'` (line 7) both describe the archive phase as writing files. But the archive phase generates content into `archiveFiles[i].content` and returns it to the caller. No `Write` tool call or agent call writes the files. The `return.instructions` string tells the caller to write, but the instruction string is not a reliable mechanism.

Finding: The `Archive` phase produces `archiveFiles` in memory and does not write them. Any caller that doesn't implement the write step silently succeeds while producing no output files.

Fix option A: Add a write agent at the end of the archive phase:
```javascript
await parallel(validFiles.map(f => () =>
  agent(`Write this content to ${archivePath}/${f.slug}.md using the Write tool: ${f.content}`,
    { label: `write:${f.slug}`, phase: 'Archive' })))
```

Fix option B: Rename the phase to 'Generate archive content' and update meta to reflect that writing is the caller's responsibility, and document the caller contract explicitly.

Fix option A is preferred — eliminates the caller contract gap.

Counter-check: Is delegating to the caller a deliberate design? The caller must always write anyway — the script returns the content. But the phase label actively misrepresents this.

### **P2 reliability — semantic dedup gap acknowledged but unmitigated**

Severity: P2 — inflated claim counts from multiple agents finding the same insight under different keys; distorts synthesis grouping thresholds
Category: reliability
Blocking: NO

Root cause: Lines 155–173 dedup claims by exact key match only. The comment at line 158 explicitly states: "semantic duplicates with different keys are NOT caught here." The mitigation claimed is: "synthesis grouping handles this." But synthesis grouping (4-claim threshold) counts by key count, not by distinct concepts — so 3 agents returning the same insight under different keys produces 3 claims in a group, potentially pushing that group above the 4-claim HIGH threshold without additional independent evidence.

Finding: Multiple parallel agents (e.g., 3 agents running the same research wave) can produce semantic duplicates that inflate claim counts and artificially raise synthesis confidence tiers.

Fix: Add a semantic-dedup hint at the boundary between research and synthesis:
```javascript
log(`Semantic dedup note: ${allClaims.length} unique keys across ${angles.length} agents. Semantic duplicates (same insight, different keys) are NOT removed here — check for clustering in synthesis grouping.`)
```
And add a heuristic: if two claims share more than 3 words in their key and come from different agents, flag them for manual review.

Counter-check: Is the synthesis grouping actually robust to this? The synthesis procedure groups by `group:` field, not key — if agents assign the same group, all duplicates land together and inflate that group's count. Confirmed as a real gap.

### **P3 schema — CLAIM_SCHEMA declares required fields that the archive assembly handles as optional**

Severity: P3 — inconsistency between schema contract and implementation
Category: maintainability
Blocking: NO

Root cause: `CLAIM_SCHEMA.required` (line 52-53) includes `saturated` and `queriesRun`. But the archive assembly at line 259 uses `result.queriesRun || []` — treating it as optional. Schema says required; code says optional. If a future refactor removes the defensive `|| []`, it will break.

Finding: Schema contract and defensive handling are inconsistent. Pick one: either make the schema enforce the fields (remove the fallback) or remove them from `required`.

Fix: If these fields are truly required, remove the `|| []` fallback and add a validation check:
```javascript
if (!result.queriesRun) throw new Error(`angle ${angle.title}: queriesRun missing in schema response`)
```

P3 (minor, non-blocking).
